# Product Pricing and Backlog Notes

## Принципи ціноутворення

### 1. Ціноутворення — окремий модуль
Ціна не повинна жорстко перезаписуватись під час імпорту прайсу постачальника.

Логічний ланцюг:
1. Імпорт прайсу / приходна накладна
2. Збереження закупівельних цін постачальника
3. Окремий модуль `Встановлення цін`
4. Після проведення документа `Встановлення цін` — запис продажної ціни в картку товару

### 2. Картка товару
У картці товару зберігається актуальна продажна ціна і валюта продажу.

### 3. Джерела ціни
Продажна ціна може формуватись:
- з ціни постачальника з прайсу
- з окремої ручної націнки
- з націнки по категорії
- з націнки, заданої постачальником
- вручну оператором

### 4. UI/UX логіка ціноутворення
У модулі `Встановлення цін` потрібно передбачити режими:
- не змінювати ціни
- оновити тільки нові товари
- оновити всі вибрані товари
- оновити тільки якщо змінилась закупівельна ціна
- оновити тільки якщо змінилась розрахункова продажна ціна

Також потрібно передбачити джерело формули:
- брати рекомендовану ціну з прайсу
- націнка по категорії
- націнка по бренду
- націнка по постачальнику
- ручна націнка
- ручна ціна

### 5. Імпорт прайсу
Імпорт не змінює напряму продажні ціни в картці товару.
Імпорт оновлює:
- дані товару постачальника
- закупівельну ціну
- валюту
- курс на дату
- наявність
- службові поля синхронізації

## Поведінка при повторному імпорті товару
Потрібно дати користувачу вибір режиму:
- не оновлювати існуючі записи
- оновлювати тільки службові поля
- оновлювати закупівельну ціну, якщо вона змінилась
- оновлювати всі дозволені поля

Продажні ціни картки товару напряму не перезаписуються імпортом.

## Відкладений модуль
### Контроль зниклих товарів у прайсах постачальників
Цей модуль реалізується пізніше окремо.

Потрібно буде:
- фіксувати дату останньої появи товару у прайсі
- фіксувати кількість імпортів, де товар був відсутній
- вивести окремий системний інтерфейс контролю
- дозволити правила архівації / приховування / сповіщення

Статус: `backlog / deferred`
